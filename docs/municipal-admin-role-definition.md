# Municipal Admin Role — Definition

**Bantayog Alert — Disaster Reporting Platform**
**Province of Camarines Norte, Philippines**

---

## Who Are Municipal Admins?

**Municipal Admins** are the **operational commanders** for disaster response within their assigned municipality. They are the bridge between citizens reporting incidents and responders taking action.

**Official Capacity:** MDRRMO (Municipal Disaster Risk Reduction and Management Office) staff, municipal emergency coordinators, or designated government officials.

---

## Primary Responsibilities

### 1. Report Verification & Triage
- Review citizen-submitted reports
- Verify legitimacy (not spam, not prank, not duplicate)
- Classify incident type (flood, landslide, fire, etc.)
- Assess severity level (Low, Medium, High)
- Reject invalid reports with reason

### 2. Responder Dispatch & Coordination
- Select appropriate responder type(s) for incident
- Choose specific responders based on availability and location
- Dispatch responders to incidents (opt-in acceptance model)
- Monitor responder acceptance and response times
- Reassign if responder declines or doesn't respond

### 3. Incident Management
- Monitor all active incidents in municipality
- Track responder status (en route, on scene, resolved)
- Coordinate multiple responders on same incident
- Make operational decisions during response
- Request backup or additional resources as needed
- Close incidents when resolved

### 4. Citizen Communication
- Respond to citizen questions about their reports
- Request clarification or additional information
- Provide updates on incident status
- Share official warnings and alerts

### 5. Cross-Municipality Coordination (Limited)
- Communicate with adjacent municipalities for incidents near borders
- Request mutual aid when needed
- Escalate to provincial level when beyond municipal capacity

## What Municipal Admins CAN Do

| Action | Details |
|--------|---------|
| **View reports** | All reports in their municipality (verified + pending) |
| **Verify reports** | Mark as verified, rejected, or need more info |
| **Classify incidents** | Set incident type (flood, fire, landslide, etc.) |
| **Assess severity** | Set severity level (Low, Medium, High) |
| **Dispatch responders** | Select and dispatch responders in their municipality |
| **Monitor responders** | See all responders' status, availability, location |
| **View responder location** | Real-time tracking during active dispatch |
| **Communicate with responders** | Two-way messaging during incidents |
| **Communicate with citizens** | Follow-up on reports, request clarification |
| **Request backup** | Ask for additional responders or resources |
| **Escalate to provincial** | Request provincial resources when needed |
| **View municipality analytics** | Response times, incident types, trends, **anonymized provincial comparisons** |
| **Create alerts** | Send official warnings to citizens in municipality |
| **Close incidents** | Mark resolved incidents as closed |
| **View incident history** | Full timeline and audit trail |
| **See team coordination** | View all responders on same incident |

---

## What Municipal Admins CANNOT Do

| Action | Why |
|--------|-----|
| **View reports outside municipality** | Jurisdiction boundary (**except border incidents shared with adjacent municipality**) |
| **Manage responders outside municipality** | Jurisdiction boundary |
| **Promote users to admin/superadmin** | Provincial Superadmin only |
| **Access other municipalities' analytics** | Privacy + jurisdiction |
| **Delete verified reports** | Data integrity (public record) |
| **Modify citizen reports** | Preserve original report data |
| **Bypass responder opt-in** | Responders must accept dispatches |
| **Directly edit responder notes** | Audit trail preservation |
| **Access system-wide analytics** | Provincial Superadmin only |
| **Modify security rules** | Provincial Superadmin only |
| **Manage other admins** | Provincial Superadmin only |

## Reporting Line & Authority

### Municipal Admins Report To:
- **Provincial Superadmin** (PDRRMO level)
- Can escalate incidents beyond municipal capacity
- Can request additional resources
- Can request inter-municipality coordination

### Municipal Admins Supervise:
- **Responders** in their municipality (dispatch, coordinate, monitor)
- **Citizen reports** (verify, triage, classify)
- **Incidents** within their municipality (lifecycle management)

---

## Decision-Making Authority

### Autonomous Decisions (No Approval Needed):
- Verify or reject any report in municipality (including **permanent rejection** of spam/pranks)
- Classify incident type and severity (**system suggests severity** based on citizen answers, admin confirms)
- Dispatch any responder in municipality
- Reassign responder **after communicating** with them (must message or call first)
- Request backup within municipality
- Send alerts to citizens in municipality (**including emergency alerts** like evacuation warnings)
- Close resolved incidents

### Decisions Requiring Coordination:
- **Cross-municipality incidents (border areas)** → **Both municipalities see the incident**, coordinate response
- **Provincial escalation** → Request approval from Provincial Superadmin
- **Mutual aid requests** → Coordinate with receiving municipality
- **Responder reassignment** → Must communicate with responder first (message or call) before reassigning

### Decisions Beyond Authority:
- **Promote/demote admins** → Provincial Superadmin only
- **System-wide policy changes** → Provincial Superadmin only
- **Inter-municipality dispatches** → Requires coordination + approval

## Key Characteristics

### Operational Focus
- **24/7 responsibility** (disasters don't follow office hours)
- **Time-critical decisions** (every minute matters)
- **High-stress environment** (life-or-death decisions)
- **Resource constraints** (limited responders, limited equipment)
- **Multiple concurrent incidents** (manage 5+ incidents at once)

### Scope Limitations
- **Municipality-bound** (cannot see outside their area)
- **Responder-dependent** (cannot force responders to accept)
- **Information-dependent** (rely on citizen reports for situational awareness)
- **Coordination-heavy** (must balance multiple responders, incidents, priorities)

### Success Metrics
- **Response time** (from report to dispatch)
- **Verification accuracy** (minimize false positives/negatives)
- **Responder utilization** (minimize idle time, avoid over-dispatching)
- **Incident resolution rate** (close incidents promptly)
- **Citizen satisfaction** (timely, accurate responses)

## Typical User Profile

### Role Types:
- **MDRRMO Head** (Primary admin, full authority)
- **MDRRMO Staff** (Secondary admin, delegated authority)
- **Municipal Emergency Coordinator** (Designated by mayor)
- **Station Commanders** (Fire, police, medical - with admin rights)

### Technical Literacy:
- Comfortable with web and mobile apps
- Can manage multiple concurrent tasks
- Understands maps and location data
- Can type quickly (communication critical)

### Work Environment:
- **Office-based** (MDRRMO operations center - **desktop primary**)
- **Map-centric workflow** (map always visible, all actions accessible without looking away)
- Field-deployed (during disasters - mobile secondary)
- On-call rotation (after-hours, weekends)
- Multi-screen setup (map + incidents + responders common)

## Accountability & Oversight

### Municipal Admins Are Accountable To:
- **Provincial Superadmin** (direct supervisor)
- **Local Chief Executive** (Mayor/Governor)
- **Citizens** (transparent operations)
- **Responders** (fair dispatch practices)

### Oversight Mechanisms:
- All actions logged (who did what, when)
- Audit trail for every incident
- Performance metrics (response times, accuracy)
- Citizen feedback (report satisfaction)
- Responder feedback (dispatch fairness)

## Required Qualifications

### Minimum Requirements:
- **Employment:** MDRRMO staff or designated municipal official
- **Training:** Basic incident management training
- **Availability:** On-call rotation (24/7 coverage)
- **Account:** Verified by Provincial Superadmin

### Preferred Qualifications:
- **Experience:** Emergency response or disaster management
- **Certification:** Incident Command System (ICS) training
- **Knowledge:** Familiar with municipality (geography, resources)
- **Skills:** Crisis communication, decision-making under pressure

## Day-to-Day Activities (Typical)

### During Normal Operations:
- Monitor incoming citizen reports
- Verify and triage new reports (10-30 min per report)
- Dispatch responders to verified incidents
- Monitor active incidents (5-20 at a time)
- Communicate with responders (updates, backup requests)
- Respond to citizen inquiries
- Update incident statuses
- Close resolved incidents

### During Major Disasters:
- Surge of reports (50+ in 1 hour)
- Prioritize life-threatening incidents first
- Coordinate multiple responder teams
- Request provincial resources
- Send emergency alerts to citizens
- Manage evacuation centers
- Coordinate with adjacent municipalities
- Provide situation reports to provincial superadmin

---

## Pain Points & Challenges

### Common Challenges:
1. **Information Overload** - Too many reports during disasters
2. **Responder Shortages** - Not enough responders for all incidents
3. **False Reports** - Citizens pranking or mistaken reports
4. **Poor Citizen Info** - Vague reports, bad locations, no photos
5. **Responder Non-Response** - Responder doesn't accept dispatch
6. **Resource Conflicts** - Multiple high-priority incidents, limited responders
7. **Cross-Boundary Issues** - Incidents near municipal borders (**solved: both municipalities see it**)
8. **Citizen Expectations** - Citizens want immediate response, limited resources
9. **Map Fragmentation** - Multiple windows, losing situational awareness (**solved: map always visible**)

### System Helps With:
- **Prioritization tools** - Severity sorting, responder availability, **system-suggested severity**
- **Quick verification** - Photos, location, citizen history
- **Responder status** - Real-time availability, location tracking
- **One-tap actions** - Fast dispatch, status updates
- **Automated notifications** - Citizens updated automatically
- **Escalation paths** - Clear process for provincial help
- **Map-centric design** - All actions accessible without taking eyes off the map
- **Border incident sharing** - Adjacent municipalities both see border incidents
- **Anonymized analytics** - Compare performance without exposing sensitive data

## Critical Design Constraint: Map-Centric Interface

**NON-NEGOTIABLE:** Municipal admins work at desktop in operations centers. They must see the map **at all times**. Every action must be accessible **without taking eyes off the map**.

### Design Philosophy:
```
┌─────────────────────────────────────────────────────────────────┐
│                    [MUNICIPAL MAP]                            │
│                  (ALWAYS VISIBLE - FULL SCREEN)                 │
│                                                                  │
│   ┌────────────────────────────────────────────────────────┐    │
│   │  📍 Active incidents (color-coded by severity)        │    │
│   │  🔵 Responder locations (real-time)                   │    │
│   │  🏠 Municipal boundaries                               │    │
│   │  🛣️ Roads, landmarks, geography                        │    │
│   └────────────────────────────────────────────────────────┘    │
│                                                                  │
│   [OVERLAYS - toggle without looking away]                      │
│   ☑ Active incidents only                                       │
│   ☑ Responder locations                                         │
│   ☑ Heat map (last 24h)                                         │
│   ☑ Border incidents (shared with adjacent municipality)        │
│                                                                  │
│   [QUICK ACTIONS - always visible at edges]                     │
│   [📋 Pending: 3]  [🚒 Available: 5]  [⚠️ Alerts: 0]            │
│                                                                  │
│   [CONTEXT PANEL - slides in from right, map stays visible]     │
│   → Click incident → Details slide in (map still visible behind) │
│   → Click responder → Status slides in (map still visible behind) │
│   → Click report → Verification slides in (map visible behind)   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### Key Principles:
1. **Map is always the background** - Never covered, never hidden
2. **Overlays are transparent** - Toggleable without blocking view
3. **Panels slide in/out** - Context appears, map remains visible
4. **Quick actions at edges** - Don't obscure center of map
5. **Keyboard shortcuts** - Everything accessible without mouse
6. **Multi-screen aware** - Can expand to second monitor for map + panels

### What This Means:
- NO full-screen modals that cover the map
- NO navigation away from map view
- NO popups that block map interaction
- NO separate "map view" and "list view" - everything happens on the map
- ALL actions happen ON the map or in overlay panels
