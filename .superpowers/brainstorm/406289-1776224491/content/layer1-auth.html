Secure municipality-bound access control

Current State (What You Have)

  • Provincial superadmin can create municipal admin accounts

What Needs to Be Added

1. Custom Claims with Municipality

Cloud Function: setMunicipalClaims

  • Triggered on municipal admin login
  • Sets custom claims: {role: 'municipal_admin', municipality: 'Daet'}
  • Claims included in Firebase ID token
  • Available in Firestore rules as request.auth.token.municipality
Why: Server-enforced municipality binding. Can't be spoofed by client.

2. Firestore Rules Enforcement

Security Rules: municipality-bound access

  • Reports: resource.data.municipality == request.auth.token.municipality
  • Responders: Same municipality check
  • Incidents: Same municipality check
  • Read operations: Filter by municipality
  • Write operations: Verify municipality match
Why: Database-level security. Even if client is compromised, data stays siloed.

3. Client-Side Route Guards

React Router: Protected Routes

  • MunicipalAdminRoute wrapper component
  • Checks: User is authenticated + role is municipal_admin
  • Loads municipality from user profile
  • Redirects if not authorized
  • Passes municipality to child components
Why: Good UX (friendly errors) + defense-in-depth. Fails fast if not authorized.

4. Auth Hook for Municipality

Hook: useMunicipalAdminAuth

  • Returns: {municipality, user, isLoading, error}
  • Loads municipality from user profile
  • Validates custom claims
  • Handles loading states
  • Used by all municipal admin components
Why: Single source of truth for auth state. Easy to test and mock.
