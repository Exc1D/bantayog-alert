Data Services Architecture

1. Municipal Admin Firestore Services

Municipality-aware data access methods

// Reports in admin's municipality getMunicipalityReports(municipality) getPendingReports(municipality) getReportById(reportId, municipality) updateReportStatus(reportId, status, municipality)
  • All queries include municipality filter
  • Server-side filtering (Firestore queries)
  • Client receives only their municipality's data
  • Type-safe with TypeScript

2. TanStack Query Hooks

Real-time data fetching with caching

// Hook example useMunicipalReports(municipality) { return useQuery({ queryKey: ['reports', municipality], queryFn: () => getMunicipalityReports(municipality), refetchInterval: 30000, // 30s auto-refresh }) }
  • Automatic caching reduces Firestore reads
  • Background refetching keeps data fresh
  • Optimistic updates for better UX
  • Error handling built-in

3. Real-Time Subscriptions

Firestore onSnapshot for instant updates

  • Incident status changes (verified → dispatched → resolved)
  • Responder location updates (every 30s)
  • New reports (appear immediately in queue)
  • Dispatch acknowledgments (responder accepts/declines)
  • Use onSnapshot + queryClient.invalidateQueries()
Pattern: Real-time for critical data, polling for non-critical

4. Error Handling & Retry Logic

Resilient data fetching for disaster scenarios

  • TanStack Query built-in retry (exponential backoff)
  • Firestore offline support (data persists locally)
  • Network error detection (show warning banner)
  • Stale data display (better than blank screen)
  • Manual retry buttons on critical errors
Philosophy: Degrade gracefully, never go blank during disaster

Real-Time Data Flow

How Data Flows Through the System
📥 Data In (Real-Time)
  • Firestore onSnapshot → New report submitted
  • → Triggers queryClient.invalidateQueries()
  • → TanStack Query refetches data
  • → Components re-render with new data
  • → Map shows new pin immediately
📤 Data Out (Optimistic Updates)
  • Admin verifies report → Update UI immediately
  • → TanStack Query setQueryData() (optimistic)
  • → Firestore write happens in background
  • → If write fails, rollback UI + show error
  • → User sees instant feedback
💡 Key Insight: Real-time for critical events (new incidents, status changes), polling for non-critical (responder locations every 30s). Best of both worlds.

Data Query Strategy

✅ TanStack Query Benefits

  • Automatic caching - Reduces Firestore reads (cost + performance)
  • Background refetch - Data stays fresh without manual refresh
  • Stale-while-revalidate - Show cached data immediately, update in background
  • Optimistic updates - Instant UI feedback on user actions
  • Error boundaries - Graceful degradation on failures

⚠️ Implementation Considerations

  • Query keys must include municipality (cache isolation)
  • Real-time + polling mix needs coordination
  • Large datasets need pagination (don't fetch all incidents)
  • Cache invalidation strategy for cross-component updates
  • Testing requires mocking TanStack Query and Firestore
