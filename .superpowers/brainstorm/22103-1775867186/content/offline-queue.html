Offline Queue & Sync Behavior

How should reports queue and sync when connection is poor?

A

Auto-Sync Transparently

Queue in background, sync when connection returns. Show subtle indicator. User doesn't think about it.

BEST: Set-and-forget, works like magic

B

Manual Sync Control

Show queue count, user taps "Sync Now". More control, more awareness, more cognitive load.

GOOD: User control, BAD: More to think about

C

Hybrid: Auto + Manual Option

Auto-syncs in background BUT shows queue indicator with manual sync button for impatient users.

BALANCED: Magic + control when wanted

Offline Behavior Breakdown

// When offline:
1. User submits report
2. Detect navigator.onLine === false
3. Save to IndexedDB 'offline-queue' store
4. Show: "Saved locally. Will sync when online."
5. Add queue indicator to header
// User continues using app (map from cache)
// When connection returns:
1. Detect navigator.onLine === true
2. Auto-sync all queued reports in background
3. Update IndexedDB: mark as 'synced'
4. Show success toast: "2 reports synced!"
5. Remove queue indicator from header
// User can tap "Sync Now" anytime if impatient

IndexedDB Schema

Store Purpose Key
offline-queue Queued reports awaiting sync timestamp + random
report-history All submitted reports (synced + pending) reportId
map-cache Cached map tiles for offline viewing tileUrl + zoom
feed-cache Cached incident feed 'last-fetch'

Conflict Resolution

⚠️ Edge Case: Multiple reports, same incident
If user submits report offline, and another user reports same incident (which gets verified before user comes online), what happens?
Solution: Server checks for duplicates on sync. If duplicate detected within 500m and 30 minutes, merge reports and notify user: "Your report was merged with an existing incident. Thanks for confirming!"

Key Principles

  • Queue in IndexedDB - Survives app restart, persists across sessions
  • Auto-sync on connection - Service worker listens for online event
  • Manual sync option - For impatient users, "Sync Now" button
  • Clear feedback - Queue count, success message, error handling
  • No blocking - User can continue using app (cached map/feed)
  • Deduplication - Server checks for merges on sync
